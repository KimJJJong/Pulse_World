# 전투 비트 스케줄링 수정 보고서

작성일: 2026-04-28

## 문제 요약

기존 호스트 전투 로직은 같은 비트 안에서도 처리 시점이 둘로 갈라져 있었습니다.

- `Move` 입력은 호스트가 패킷을 받는 즉시 적용됨
- 스킬 `Damage` 이벤트는 별도 비트 루프에서 나중에 적용됨

이 구조 때문에 같은 비트에서 다음 현상이 간헐적으로 발생할 수 있었습니다.

- 경고 타일에서 다음 턴에 벗어났는데도 데미지를 맞음
- 안전 타일에서 경고 타일로 들어왔는데도 데미지를 맞지 않음

원인은 "같은 비트의 최종 위치"를 기준으로 데미지를 보지 않고, 프레임 순서와 패킷 도착 순서에 따라 이동과 데미지가 섞여 처리되었기 때문입니다.

## 기존 문제 흐름

기존 `P2PHostController`는 다음처럼 동작했습니다.

1. 현재 비트에 도착한 입력을 바로 처리함
2. `Move`는 즉시 위치를 바꿈
3. `Skill`은 바로 스케줄에 넣은 뒤, 어떤 경우에는 같은 프레임에 `ProcessSkillsAtBeat(currentBeat)`까지 실행함
4. 이후 비트 루프에서 `Damage`를 처리함

이 구조에서는 같은 비트라도 다음이 보장되지 않았습니다.

- 이동이 데미지보다 먼저 확정되는지
- judge window가 닫힌 뒤 최종 입력 상태만 반영되는지
- 이미 확정된 지난 비트 입력이 뒤늦게 월드 상태를 바꾸지 않는지

## 수정 후 확정 스케줄

수정 후 기준은 "현재 비트"가 아니라 "judge window가 끝난 마지막 확정 비트"입니다.

비트 `B`는 아래 순서로 처리됩니다.

1. 현재 비트 시점:
   공격/스킬 입력은 즉시 받아서 시각 연출과 스킬 예약을 시작함
2. 확정 시점:
   `beatTime(B) + judgeWindowMs`가 지난 뒤에만 비트 `B`를 실제 판정함
3. 비트 `B` 판정 순서:
   같은 비트의 `Move` 명령 확정
4. 비트 `B` 스킬 이벤트 순서:
   기존 우선순위대로 `Move -> Warning -> InputLock -> Damage`
5. 지난 비트 입력:
   이미 확정 완료된 비트보다 늦게 도착한 입력은 폐기

즉, 데미지는 항상 "그 비트의 이동이 모두 반영된 뒤의 최종 위치"를 기준으로 들어갑니다.

## 코드 변경점

대상 파일:

- `Client/Assets/0.MainProject/04_Runtime/Game/Managers/P2PHostController.cs`

핵심 변경:

- `GetLastResolvedBeat()` 추가
  - `serverNow - judgeWindowMs` 기준으로 더 이상 입력이 뒤집히지 않는 마지막 비트를 계산
- `Update()` 변경
  - 즉시 처리 단계와 확정 처리 단계를 분리
- `ProcessImmediateQueuedCommands()` 추가
  - `Attack/Skill`은 현재 비트에서 즉시 예약
- `ProcessQueuedCommandsAtBeat()` 변경
  - `Move`는 확정 비트에서만 처리
- 늦게 도착한 입력 방어 추가
  - `executeBeat <= _lastProcessedBeat` 인 입력은 폐기
- 즉시 스킬 처리 시의 재진입 제거
  - `_activeSkills.Add(scheduled);` 후 바로 `ProcessSkillsAtBeat(currentBeat)`를 호출하지 않도록 수정

## 기대 효과

- 경고 타일에서 다음 턴에 빠져나오면 데미지를 맞지 않음
- 안전 타일에서 경고 타일로 들어오면 데미지를 맞음
- 같은 비트에서 이동과 데미지의 선후가 프레임 타이밍에 흔들리지 않음
- 지난 비트 입력이 뒤늦게 판정을 오염시키지 않음

## 확인 권장 시나리오

1. 경고 타일 위에서 다음 비트에 이동 입력
   - 결과: 이동 후 안전 타일이면 무피해
2. 안전 타일에서 다음 비트에 경고 타일로 이동
   - 결과: 데미지 적용
3. 2인 P2P에서 게스트가 judge window 끝부분에 이동 입력
   - 결과: 같은 비트 데미지보다 먼저 위치가 확정되어 일관된 결과
4. 이미 지난 비트 입력을 인위적으로 늦게 전달
   - 결과: 호스트가 입력을 폐기
